# newton-cli 这个项目到底在干嘛

## 一句话解释

**让 AI 能用打命令的方式玩物理仿真，而不用写代码。**

## 背景

英伟达最近发了一个叫 **Newton** 的物理引擎。它能模拟真实世界的物理 ——
球会滚、布料会飘、机器人会走路、软体会变形。

问题：它是一个 Python 库。要用它，你得会写 Python 代码。

目标：让 AI（比如 Claude、GPT）能通过打一两行命令就驱动 Newton，
而不用写一大坨 Python。就像 AI 已经会用 `ls` `git` 这种命令行工具一样，
现在让它学会用 Newton。

我们的产品叫 `newton-cli`，就是给 Newton 套了一层"命令行外壳"。

---

## 思路：让例子当考官

Newton 自己带了**几十个示例**（examples）：单摆、一堆形状掉地上、
四足机器人走路、布料下垂、软体悬挂……每个示例的末尾都写着一段
"验证代码"——检查模拟结果是不是对的。

我们的办法是：

> **把每个示例都变成一道"考题"。如果 newton-cli 能通过命令把这个示例
> 的物理场景复现出来，并且能通过示例自己的验证，就算这道题过了。**

哪个示例考不过，说明 CLI 少了某种能力，就去补那种能力。
哪道题考过了，就记下来继续下一道。

这样做的好处：
- 不会做多余的功能（只有示例需要的才做）
- 有明确的通过标准（示例自己的验证）
- 进度可量化（X / Y 道题通过了）

---

## 核心发明：把模型写成"菜谱"

Newton 的 Python 代码里要搭一个物理场景大概长这样：

```python
builder = newton.ModelBuilder()
link_0 = builder.add_link()
builder.add_shape_box(link_0, hx=1.0, hy=0.1, hz=0.1)
builder.add_joint_revolute(parent=-1, child=link_0, axis=[0,1,0], ...)
...
```

一连串 Python 方法调用。AI 不好写这种代码，也没法通过命令一次一个地执行。

**我们发明的办法**：让用户写一个 **JSON 格式的"菜谱"**，描述这些步骤：

```json
{
  "ops": [
    {"op": "add_link"},
    {"op": "add_shape_box", "args": {"body": 0, "hx": 1.0, "hy": 0.1, "hz": 0.1}},
    {"op": "add_joint_revolute", "args": {"parent": -1, "child": 0, "axis": [0,1,0]}}
  ]
}
```

然后跑两条命令就行：

```bash
# 从菜谱搭建模型
newton-cli model build --recipe 菜谱.json --out 模型.json

# 跑仿真
newton-cli sim run --model 模型.json --solver SolverXPBD \
                   --num-frames 100 --out 结果.npz
```

**意外的好处**：这个"菜谱"本身就是模型的保存格式。
Newton 原本没有现成的模型文件格式，而我们一下就有了 ——
因为菜谱重新执行一遍就能得到完全一样的模型。

## 遇到的坑（也都解决了）

### 坑 1：Newton 一启动就往屏幕打字
Newton 底层的 Warp 库初始化时会打印一堆设备信息，把我们的 JSON 输出
污染了。**解决**：在最早的位置拦截住这些输出。

### 坑 2：MuJoCo 求解器在 GPU 上有微小的不确定性
同样的输入，在两个 Python 进程里跑，结果会差一点点（float 级别）。
简单场景差 0.0001，复杂机器人差 0.1。**解决**：分级验收 ——
简单场景要求精确匹配，复杂场景只要求"物理上合理"（没炸、没掉出地图）。

### 坑 3：有些示例用 Python 硬编码控制每一步
比如传送带例子里，作者写了一个 Warp 内核每一步都调用一次来驱动皮带。
这种"每一步跑一段 Python"是没法用静态 JSON 菜谱表达的。
**解决**：承认这是边界，把这类示例标成"暂缓" ——
总共暂缓了 5 个示例，都有清晰的理由。

### 坑 4：机器人需要特殊的稳定化技巧
ANYmal 四足机器人必须额外给每个身体加一点"转动惯量"才能稳定站立，
这是物理仿真的老技巧。**解决**：加了一个专门的"菜谱动作"叫
`apply_body_inertia_diagonal`。

## 关键判断：要不要让菜谱里夹 Python？

现在 `newton-cli` 能跑**任何只需要"搭场景 + 按标准步骤仿真"的示例**，
哪怕场景再复杂。

**但是**，有些示例需要在每一步仿真中间插一段 Python 逻辑
（比如跑一个 PyTorch 神经网络策略来控制机器人走路）。
这类示例占 Newton 示例库的一小部分，但很重要。

要支持它们就得让用户在菜谱里夹 Python 代码路径，
这就**破坏了"AI 只写 JSON"的承诺**。

**当前决定**：不做这个妥协。先把"纯 JSON 能搞定的"都搞定，
等真正需要的时候再回头讨论。

## 给不看代码的人的最终感受

这次做的事情，本质上是"把一个给程序员用的 Python 库，
改造成可以用打命令方式驱动的工具"。

难点不在代码量，而在**每一步都要做对决策**：
- 要不要用某个第三方工具？（讨论后没用，手写）
- 模型文件应该是什么格式？（发明了"菜谱即模型"）
- 测试通过的标准是什么？（分三档）
- 哪些示例能做、哪些要放弃？（有明确边界）

你可以把它理解为：我们给 Newton 做了一个"AI 翻译层" ——
AI 只需要写 JSON，就能驱动原本需要写 Python 才能用的物理引擎。
